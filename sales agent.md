# AI Chat 产品变现架构思考

## 1. 背景

近期 AI 产品高速发展，目前可见的、流量增长明显的产品形态主要有两种：

- **对话类**：以豆包为代表
- **AI 搜**：以 Google AI Overview 为代表

市场上普遍认为 **2026 年为 AI 变现元年**。但对于 AI 两种形态的具体变现方式，大部分思路仍比较固化，本文尝试提出一些新的思考。

## 2. 现状：各家变现思路

### 2.1 Chat / AI Search 变现方式概览

**Chat 产品（本次重点讨论）**

- 订阅
- 品牌广告
- 抽佣（生活服务、电商）+ 效果广告

**AI Mode 产品**

- Google：商品广告为主（已完善）
- 百度：效果广告为主（已完善、接近天花板）

### 2.2 竞品现状

#### 2.2.1 ChatGPT 现状

ChatGPT 是目前变现链路最完整的标杆。截至 2026 年 5 月初：

- **端到端打通**：已自建投放平台
- **流量侧**：实现意图识别推荐广告
- **投放侧**：
  - 计费方式：初期是 CPM 计费，目前已有 CPC 计费；尚无成熟的归因基建
  - 转化目标和报告类似传统广告

#### 2.2.2 其他产品

- Google Shopping / Lens（待调研）
- 亚马逊购物助手（待调研）
- 淘宝 AI 导购（待调研）

### 2.3 三种产品模式对比

| 维度 | Feed 广告 | 搜索广告 | AI Chat 广告 |
| --- | --- | --- | --- |
| **广告请求** | 用户向下滑动 | **Query 驱动** | **意图驱动**：识别到商业意图才下发请求 |
| **意图信息密度** | 低：兴趣标签，激发为主 | 高：单一维度 | **超高**：自然语言多维度约束，场景 + 偏好 + 限制 |
| **广告库存** | Adload | 上方位 | **意图强度 × Adload** |
| **广告位结构** | 独立隔离位 | 独立隔离位 | **嵌入式？** 商品卡融合在 AI 回答正文里 |
| **用户决策状态** | 发现 / 激情驱动 | 需求明确，主动寻找 | 需求复杂模糊，需要 AI 辅助决策 |

### 2.4 AI Chat 和搜索广告的相似点

- **意图为核心**：搜索广告有长时间的意图积累，比如意图 N 阶段等。
- **对话类和“搜索 + 在客户落地页内对话”本质上是一件事情**。
- AI Chat 场景对比搜索场景，“结果”本身要考虑 **Session + 用户信息**；搜索则更聚焦于 Query 本身。

## 3. 核心判断

### 3.1 流量判断：AI Chat 是否带来新的流量

AI Chat 会切走很多搜索流量，但由于满足程度更好，也会新增一些流量。AI Chat 不会完全代替搜索和 Feed。

### 3.2 预算判断

#### 3.2.1 是否带来新的预算

会。由于用户满足度提升，更好、更契合的广告会得到展现曝光和消费。这是**增量预算**，不是左手倒右手。

#### 3.2.2 预算类型：线索 = 电商 > 下载类

不同行业差异极大，**重点在线索行业和电商**。其中，**线索行业想象力更强，App 下载类更弱**。

- **品类匹配度**：线索行业（教育、医美、装修、汽车、法律、保险、医疗咨询、家政）的共性是“决策周期长 + 信息严重不对称 + 用户渴求真实建议 + 单条线索价值高”，比较匹配 AI Chat 场景。
- **商业模式优势**：线索行业按 CPL / CPA 计费，LLM 能直接介入“询问 — 筛选 — 促成留资”的整个漏斗，不像电商需要打通商品库、价格、库存、履约等一长串后台。技术上，线索类 AI Agent 的 **MVP 成本远低于电商**。
- **风险点**：
  - 广告主认知和基建远远不够，这部分在传统广告时期就已经存在，微信的想象空间更大。
  - 部分行业有幻觉带来的法律政策风险、品牌要求问题。

#### 3.2.3 与搜索、Feed 的关系

- AI Chat 不会代替搜索和 Feed，搜索会退化为**常见问题的权威解答**。
- AI 导购吃的是 **理性决策类广告主的搜索预算**（百度 SEM 那部分），**不是 Feed 的兴趣激发预算**。
- 长期看，是把外部预算搬进来，不是切自家蛋糕。

### 3.3 核心问题（待讨论）

- **AI Chat 竞价公式应该是什么？** 是否还需要一个超参公式存在？还是基于一个核心逻辑，用 LLM 推理即可？
- **AI Chat 竞价系统应该是传统方式，还是 Agent 组织方式？** 核心决策如何完成？
- **AI Chat 拍卖还遵循传统的拍卖 / 分配方式吗？**
- **广告是否干扰结果**：AI 推荐商品是否允许插广告？例如是否打广告标、是否做样式或区域划分。

### 3.4 豆包 AI 购物变现的问题观察

目前购物流程是**割裂的**：搜索、决策、对比、下单、售后非闭环。AI 对话天然有机会成为“统一接入层”。

### 3.5 前置导流策略

需要单独做一层商业意图导流，例如建议追问：

> 你是否要为老人购买宽楦的运动鞋？

### 3.6 变现效率：S 曲线

| 阶段 | 判断 | 原因 |
| --- | --- | --- |
| 初期 | 低 | 流量池小（对比 Feed 时长可能短）；曝光机会少（对比搜索，不是单屏幕 5 次）；广告主接受度初期低；技术基建需要时间；用户使用习惯需要培养。 |
| 后期 | 高 | 减少交互摩擦，转化率可提升；长尾需求承接；数据量大；预算侧粘度提升；**单次会话的信息密度更高**。后期变现高不只是“流量大了 + 习惯养成了”，更本质的是单次会话的信息密度更高。Feed 广告一次曝光大概 3-5 秒、信息维度 2-3 个；AI 对话一轮交互可以承载用户**意图 + 约束 + 偏好**等 10+ 维度信息。这种用户主动暴露的高密度意图数据，是 Feed 时代从未有过的资产。变现潜力的真正来源是这个**“用户主动暴露的意图数据池”**，而不是流量本身。 |

## 4. 整体方案概览

### 4.1 方案一：渐进式检索

#### 4.1.1 渐进式检索方式

**Turn 1**

- 用户：“想给我爸买双鞋”
- Router：商业意图识别 → 触发 Session Pool 初始化
- 【检索 L1】意图槽位“老人鞋” → 500 候选入池
- 【触发器】决策阶段 = 探索期 → 不触发广告
- AI 回复：引导式提问“您爸爸多大？平时主要在哪走路？”

**Turn 2**

- 用户：“67 岁，糖尿病脚有点肿，小区散步”
- 【检索 L2】Narrow by “67 岁 + 糖尿病脚 + 宽楦” → 80 候选
- 【触发器】决策阶段 = 探索期 → 不触发
- AI 回复：决策框架（鞋楦 / 缓震 / 鞋面 / 防滑 4 维度）

**Turn 3**

- 用户：“最在意宽楦和缓震，预算 300 以内”
- 【检索 L2】Narrow → 30 候选
- 【触发器】决策阶段 = 候选筛选 ✓；偏好维度 = 4 ✓；池子 = 30 ✓
- 强触发，展示 3 张卡片
- 【拍卖 L3】对 30 候选精排
- `actual_bid = baseline × 1.0`（筛选期 multiplier）
- `eCPM = bid × pCVR × Q × T`
- GSP 取 Top 3
- AI 回复：3 款推荐 + 优缺点 + 不适配点

**Turn 4**

- 用户：“第 1 款和第 3 款哪个防滑更好？”
- 【触发器】决策阶段 = 对比期 → 不触发新广告
- AI 回复：客观对比（不偏袒 Sponsored）

**Turn 5**

- 用户：“那我选第 1 款”
- 【触发器】决策阶段 = 收口 → 收口触发
- AI 回复：下单卡（融入对话流，但视觉隔离）

**跨 Session（30 天后）**

- 触发售后跟进：“你爸穿着舒服吗？”
- 收集反馈

#### 4.1.2 拍卖奖励函数

**主力公式：混合形式，乘法保 ROI，加法保体验**

```text
score = bid × pCVR × Q × T × ContextMul × 1000 + λ × UserValueBonus
```

- **乘法主项**：保证 ROI 最大化和质量硬阈值。
- **加法兜底项**：保证用户价值。

**乘法主项因子**

| 因子 | 含义 | 来源 |
| --- | --- | --- |
| `bid` | 商家在该意图槽位的意图达成出价 | 商家投放后台 |
| `pCVR` | `p(M → C → A)`，平台 ML 预估 | 平台模型 |
| `Q` | 意图相关性，低于阈值踢出 | LLM-as-judge |
| `T` | 信任分，低于阈值踢出 | 商家信誉模型 |
| `ContextMul` | 决策阶段乘子：探索 0.5 / 筛选 1.0 / 收口 1.5 | 对话状态识别 |

**加法兜底项 UserValueBonus（参考 Facebook）**

```text
UserValueBonus = α × 多样性奖励 + β × 用户偏好匹配 - γ × 该商家用户负反馈
```

其中，`λ` 是平衡商业项和用户价值项的全局参数，建议从 `0.1` 起步小心调。

**计费规则：GSP 二价**

```text
实付 = (下一名 score / 你的 pCVR / Q / T / ContextMul) + 0.01
```

注意：实付不除以 `UserValueBonus`。只用乘法主项做反算（这是 Google 的标准做法），加法项只影响排序，不影响计费。

### 4.2 方案二：Agent 推理

对比传统的检索流程，在 Agent 场景下，需要做一些大胆的猜想：

1. 核心部分由 Agent 编排并推理。
2. 从“排序公式”转向“Critic Agent 推理”。

传统方式：

```text
score = bid × pCVR × Q × T × ContextMul
```

Agent-native 方式：一个 Critic Agent 对每个候选用自然语言推理打分。

```text
候选 A：商家出价 10 元，但商品定位偏年轻人，对 67 岁老人不太适配。
虽然商家信任分高，但意图不匹配。综合判断：不推荐。

候选 B：商家出价 8 元，专为糖尿病脚设计，三方评测都很好。
缺点是颜色偏老气。综合判断：强推。

候选 C：商家出价 12 元（最高），但是新商家、信任度未建立。
综合判断：可以推但放第二位……
```

公式只是 Prompt 里的指导，不是 closed-form 表达式。Critic 可以根据具体场景做不同侧重，例如医美品类自动加重信任分，电商促销期自动加重价格因素。这种“权重自适应”在公式时代只能靠超参，Agent 时代可以靠推理。

## 5. 流量侧

### 5.1 产品能力与创意样式

**产品能力**

可以实现在多轮对话中进行 **“服务”的推荐** 和 **“品牌”的露出**，尤其是可以丝滑地在多轮对话中**逐步承接用户后端需求**。

**创意样式（广告长什么样）**

- 待补充

### 5.2 C 端呈现

文本、图片、富媒体均可，但核心是给用户 **控制感** 和 **信任感**。

### 5.3 信任与产品原则

#### 5.3.1 流量策略

> AI 推荐的商品必须是在 **满足用户需求的前提下做商业化**，而不是为了商业化去扭曲用户需求的满足。  
> **先帮用户选对，在选对的范围内优先展示付费商家。**

#### 5.3.2 三大机制

- **分 AB 区**：搜索广告在初期也有类似考量，如 PP、IM、PPIM 等，并标以不同的颜色框区分。
- **推荐理由可验证**：每条产品声明都能追溯到源头，如官方参数页、第三方评测、真实评论。
- **暴露缺点**：主动列出商品的不适配点，是建立信任的核心动作。

#### 5.3.3 防幻觉的产品逻辑

- 事实性信息强制走结构化商品 DB，LLM 只做表达和适配解释。
- 每一条产品声明都可点击追溯出处。
- **下单环节硬隔离**：订单生成走传统电商 API，LLM 不参与最后一步。
- **评测里专门设计幻觉率指标**：每品类、每意图分桶统计，作为模型上线门槛。

#### 5.3.4 披露策略

**是否分 AB 区**

> 建议**第一阶段不分**，让 AI 回答里直接嵌入商品卡。但**必须用视觉差异化**（背景色 / 边框 / 小标签）让用户能识别。  
> 分 AB 区会让用户像看 Feed 广告一样直接划走，违背 AI 导购“理性决策载体”的本质。

**是否披露广告**

> **必须披露，但用“推荐理由”代替“广告”**。比如商品卡上写“基于你的需求匹配（合作品牌）”，而不是写“广告”。  
> **披露 ≠ 自损**，反而是建立信任的核心动作。

**是否披露 Agent 是商家 Agent**

> **必须披露**，但同时给用户 **“切换到平台中立 Agent”的选项**。  
> 这个设计同时满足了商家诉求（让 Agent 为我代言）和用户诉求（我可以看到中立第三方意见）。

## 6. 投放侧：Agent 专属投放方式

### 6.1 投放策略

随着流量侧 AI Chat 的渗透，投放侧也会有自身的 Agent 投放方式。对应的投放策略，如出价、定向、预算、创意、报告、转化及归因是否需要变革，需要逐一讨论。

### 6.2 商家经营

商家经营需要围绕以下能力建设：

- 结构化商品
- SOP
- 品牌人设
- 知识库
- API
- 自营 Agent

## 7. 评估体系

- 模型迭代目标 = **“这一版相对上一版要好在哪、好多少、怎么验证、不能伤什么”**。
- 观察维度：模型指标、模块指标、业务指标、长期指标。
- 经典评估维度：一致性、分类、质量、相似度。
- 评估方式：标准集合人工评估、LLM 评估、对抗类 Case、自动化指标（幻觉率）。
- 和 Agent 评估相关的新增方向：
  - 模拟用户群（Simulator Agents，Agent-native 新增）
  - 反事实推理（Counterfactual）

## 8. 能力边界

### 8.1 问题

- 事实性参数（价格 / 库存 / 参数）：关键事实必须有第二信源验证，如结构化数据库 / 第三方权威 API。
- 部分品类的意图识别和判断：需要专家模型。
- 长期 LTV 观测。
- Prompt Injection。

### 8.2 原则

- 规则化程序兜底。
- 关键决策用非 LLM 触发。
- 限制 Agent 数量和深度。
- **做好 Memory**。

### 9.3 两个重点 Demo 项目

**基于山姆商品库的护肤品选购**

- 价值点：SKU 有限 + 平台背书 + 理性决策 + 用户决策疲劳痛点。
- 核心机制：列优点 + 列缺点 + 列不推荐项。

**60 岁老人的宽楦运动鞋选购**

- 价值点：搜索能力差 + 信息严重不对称 + 决策权代际转移 + 长尾 SKU。
- 核心价值：把“我爸 67 岁，糖尿病脚有点肿，平时小区散步”这种自然语言模糊需求翻译成结构化参数。
